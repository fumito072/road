## 実装上の示唆

### やってはいけないこと

- `basePath/customerName/contractNumber` のような固定3階層だけで保存先を決め打ちする
- `スキャナ/<行>/<列>/<顧客>/<商材>/<契約ID>` のように全業務共通の固定テンプレートをハードコードする
- OCR 結果だけで自動確定し、ユーザー確認なしで即アップロードする

### 必要な考え方

- OCR は「保存先の確定」ではなく「保存先候補の提案」を行う
- 保存先候補は SharePoint 上の実在フォルダ検索を通して組み立てる
- ユーザーは確認画面で `ファイル名` と `保存先フルパス` を確認し、必要に応じて修正する
- 最終アップロードはその確定後のみ行う

## 推奨する柔軟な設計

### 1. 抽出結果と保存先解決を分離する

OCR / AI が出すのは以下までに留める。

- 顧客名
- 店舗名や支店名の候補
- 商材候補
- 契約ID候補
- 書類種別
- 出力ファイル名候補

保存先パスはこれらを使って別工程で解決する。

### 2. 保存先は「テンプレート固定」ではなく「候補探索」で決める

推奨フロー:

1. タブごとにベースフォルダだけ持つ
   - 例: `スキャナ`
2. 顧客名からかなグループを算出する
   - 例: `あ/あ`
3. SharePoint 上でその配下の顧客候補フォルダを検索する
4. 顧客フォルダ配下で店舗/支店/閉店系フォルダ候補を検索する
5. その配下で商材候補を検索する
6. さらにその配下で契約ID系・年度系・商品系フォルダ候補を検索する
7. 候補を UI に出し、最終確定させる

### 3. 階層構造は設定で吸収できるようにする

業務ごとに以下のような「保存先解決ルール」を持てるようにする。

- `baseFolderPath`
- `customerFolderStrategy`
- `branchFolderStrategy`
- `productFolderStrategy`
- `leafFolderStrategy`

例:

- コラボ系は `顧客 -> 店舗 -> 商材 -> サブ分類`
- 領収証系は `顧客 -> 月別 or 利用日`
- リース系は `顧客 -> 契約ID`

この差をコード分岐ではなく設定として扱う方が安全

### 4. UI は「候補選択型」にする

確認画面で最低限必要な項目:

- 出力ファイル名
- 保存先フルパス
- その保存先が既存フォルダか新規作成か
- 顧客候補一覧
- 店舗/支店候補一覧
- 商材候補一覧
- 最下層候補一覧

ユーザーが `OK` を出すまではアップロードしない

## 現時点の実装方針案

- まず `sharepointFolderPath` を単なる完成済み文字列ではなく「候補解決結果」として扱う
- バックエンドで `resolveSharepointDestination()` のような処理を作る
- この処理は OCR 抽出値と SharePoint 実フォルダ検索結果から候補パス一覧を返す
- フロントはその候補一覧を表示し、ユーザーが 1 つ選択 or 手修正できるようにする
- confirm 後にのみ `sharepoint` API を呼ぶ

## いまの結論

- 出口 SharePoint は確認できた
- 最下層保存も確認できた
- ただし階層構造は業務や顧客によってブレる可能性がある
- そのため、ROAD は「固定パス生成」ではなく「候補探索 + 人確認」型で実装するのが妥当

## 2026-04-16 時点の整理し直し

ユーザー整理を反映すると、前提は「全業務で階層が完全にバラバラ」ではなく、以下の4業務ごとに固定の親階層と固定の配下構造を持つ、という理解が正しい。

- コラボ
- モバイル
- 現金・リース
- 電力

この前提だと、ROAD 側で最初にユーザーに業務分類を選ばせる設計は妥当である。

### 意味すること

- 4業務は UI で事前選択させる
- 選択された業務によって、探索対象の SharePoint ベースフォルダを絞り込める
- その業務配下では、下位の階層構造はその業務固有の固定ルールで扱える
- したがって「全 SharePoint を汎用探索する」より「業務別の解決ルールを持つ」方が正確

## 現時点で妥当なフロー案

1. ユーザーが UI 上で 4業務のどれかを選択する
2. OCR で会社名・契約ID・書類種別などを抽出する
3. 命名規則に従って出力ファイル名候補を作る
4. バックエンドは選択業務に対応する SharePoint ベース配下だけを探索する
5. その業務の既知の階層構造ルールに従って候補パスを解決する
6. UI に「この名前で、このパスに保存する」候補を表示する
7. ユーザーが確認して OK ならアップロードする

この方式なら、「業務ごとに構造が違う」ことと「業務内では構造が固定」の両方を矛盾なく扱える。

## 例外パターン

### 1. 新しい会社

- 顧客フォルダ自体が存在しない
- この場合は新規作成候補として扱う必要がある

### 2. 既存顧客だが、その業務が初回

- 顧客フォルダは存在する
- ただし、たとえば `コラボ` や `モバイル` の配下フォルダがまだ存在しない
- この場合も中間階層の一部を新規作成する必要がある

### 3. OCR 候補が複数にぶれる

- 会社名揺れ
- 店舗名揺れ
- 契約ID 未確定
- この場合は候補を UI に出し、人が最終選択する必要がある

## 実装方針として有効な考え方

### 1. 汎用ハードコードではなく「業務別ルール」を持つ

持つべきものは 1 個の固定テンプレートではなく、4業務ごとの保存先解決ルール。

例:

- `collaborationResolver`
- `mobileResolver`
- `leaseCashResolver`
- `powerResolver`

それぞれが、

- ベースフォルダ
- 顧客フォルダの見つけ方
- 店舗/支店フォルダの見つけ方
- 商材フォルダの見つけ方
- 最下層フォルダの見つけ方
- 足りない場合にどこまで新規作成してよいか

を知っている構造にするのがよい。

### 2. 解決処理は「探索」と「確定」を分ける

処理を 2 段に分ける。

1. `resolve`
  - OCR 結果と SharePoint 実構造から候補パスを出す
2. `confirm`
  - ユーザーが確定した候補に対して実際に保存する

この分離がないと、OCR の誤判定がそのまま誤保存に直結する。

### 3. UI に見せるべきなのは完成パス

ユーザーに見せるのは単なる顧客名ではなく、必ず完成パスにする。

表示例:

- 出力ファイル名: `株式会社ロード_CGN048549_契約書.pdf`
- 保存先: `ドキュメント/スキャナ/あ/あ/株式会社ロード/コラボ/CGN048549`
- 状態: `既存フォルダ` または `一部新規作成あり`

### 4. 新規作成は「差分だけ」に限定する

新規会社や初回業務でも、上位階層を全部自由生成するのは危険。

安全なのは以下。

- 業務ベースまでは既知ルールで決める
- 顧客候補が無ければ顧客フォルダだけ新規作成候補にする
- その下も業務ルールで必要な最小限だけを新規作成候補にする
- 作成予定階層を UI に明示する

## 推奨する最終方針

- ユーザーに最初に 4業務を選ばせる
- 4業務ごとに専用の保存先解決ルールを持つ
- OCR はフォルダを確定しない。候補の材料だけ出す
- バックエンドは選択業務のルールに従って SharePoint を探索する
- UI で完成パスを必ず確認させる
- 既存フォルダが無い場合は「どの階層を新規作成するか」を明示した上で保存する

## 現時点の意見

- この問題は「完全汎用のフォルダ探索問題」として解くより、「4業務ごとの半構造化ルール問題」として解く方が確実
- 4業務の選択を最初に持ってくるのは正しい
- 誤保存を防ぐため、確認画面は必須
- 特に新規会社 / 初回業務は自動保存ではなく、必ず新規作成予定パスを見せるべき
- したがって、最適解は「業務別 resolver + 候補パス提示 + 人確認」である

## この比較から言えること

- 4業務すべてで「業務フォルダの直下以降が完全に同じ」とは言えない
- 少なくとも今回確認したサンプルでは、以下の差がある

```text
コラボ: 業務 -> 契約IDらしきフォルダ -> ファイル
モバイル: 業務 -> 年 -> 契約IDらしきフォルダ
電力: 業務 -> 年 -> 直接ファイル
現金・リース: 業務相当 -> 年 or サブ分類 -> 作業分類 or 最下層
```

- したがって、「4業務だから配下構造も同じ」と決めるのは危険
- ただし、「4業務ごとに探索ルールを分ける」方針は引き続き有効

## 更新された実務判断

- 4業務は UI の最初の選択肢として有効
- ただし resolver は 4業務で 1 本化せず、少なくとも業務別に分けるべき
- さらに業務内でも `年フォルダ有無` や `契約ID直下かどうか` を吸収できる設計が必要
- そのため、最終的には「業務別 resolver + 実構造探索 + 人確認」が妥当

## 実装固定案: resolver 探索順 / 候補パス返却 / resolve API 仕様

ここからは実装前提で、そのまま backend / UI に落とせる粒度で固定する。

### 全 resolver 共通の考え方

resolver は以下の 3 役を持つ。

1. OCR 抽出値を正規化する
2. SharePoint 実構造から保存先候補を組み立てる
3. 候補パス一覧を UI に返す

resolver は「1 本の完成パスを即返す」のではなく、「候補一覧 + 理由 + 新規作成差分」を返す。
最終決定は必ず UI 上で人が行う。

---

## 1. 各 resolver の探索順

### 1-1. 共通前処理

全 resolver 共通で最初にやること:

1. OCR 結果の正規化
   - 会社名の全角半角揺れ除去
   - 株式会社 / (株) などの表記ゆれ吸収
   - 店舗名候補の括弧や記号の正規化
   - 契約ID候補の記号除去 / ハイフン保持方針統一
2. かなグループ推定
   - `customerName` から `あかさたな` と `あいうえお` を推定
3. 顧客フォルダ候補の収集
   - 第一候補: 推定かな階層直下
   - 第二候補: 同かな行全体
   - 第三候補: `root/search(q='<customerName>')`
4. 中間階層候補の収集
   - `本社` `支店` `営業所` `見込` `閉店` などのキーワードを抽出

この前処理の出力は各 resolver が共通で使う `customerFolderCandidates` と `branchFolderCandidates`。

### 1-2. コラボ resolver の探索順

探索対象パターンを優先順で並べる。

#### Pattern C1

```text
顧客 -> 店舗/支店/見込 -> コラボ -> 契約ID風フォルダ -> ファイル
```

#### Pattern C2

```text
顧客 -> 店舗/支店/見込 -> コラボ -> 年 -> ファイル
```

#### Pattern C3

```text
顧客 -> コラボ -> 契約ID風フォルダ -> ファイル
```

#### Pattern C4

```text
顧客 -> コラボ -> 年 -> ファイル
```

探索順:

1. 顧客候補を決める
2. 顧客配下の `店舗/支店/見込` 候補を拾う
3. その配下で `コラボ` フォルダを探す
4. `コラボ` 直下に
   - 契約ID風フォルダがあるか
   - 年フォルダがあるか
   - 直接ファイル型か
   を判定する
5. 候補パスを返す

### 1-3. モバイル resolver の探索順

#### Pattern M1

```text
顧客 -> 店舗/支店/見込 -> モバイル -> 年 -> 契約ID風フォルダ -> ファイル
```

#### Pattern M2

```text
顧客 -> 店舗/支店/見込 -> モバイル -> 年 -> ファイル
```

#### Pattern M3

```text
顧客 -> モバイル -> 年 -> 契約ID風フォルダ -> ファイル
```

探索順:

1. 顧客候補を決める
2. `店舗/支店/見込` 候補を拾う
3. `モバイル` フォルダを探す
4. `モバイル` 配下で年フォルダを探す
5. 年配下に
   - 契約ID風フォルダがあるか
   - 直接ファイル型か
   を判定する
6. 候補パスを返す

### 1-4. 電力 resolver の探索順

#### Pattern P1

```text
顧客 -> 本社/支店/見込 -> 電力 -> 年 -> ファイル
```

#### Pattern P2

```text
顧客 -> 本社/支店/見込 -> 電力 -> ファイル
```

#### Pattern P3

```text
顧客 -> 電力 -> 年 -> ファイル
```

探索順:

1. 顧客候補を決める
2. `本社/支店/見込` 候補を拾う
3. `電力` フォルダを探す
4. `電力` 直下に年フォルダがあるかを判定する
5. 年配下または `電力` 直下に直接ファイルがあるかを判定する
6. 候補パスを返す

備考:

- 電力は `契約ID風フォルダ` を標準前提にしない
- `直接ファイル保存型` を第一級の型として扱う

### 1-5. 現金・リース resolver の探索順

現金・リースは内部的にサブresolverへ分岐する。

#### Step L0: 派生区分推定

OCR / documentType / 命名規則から以下のどれかを候補化する。

- `現金`
- `リース`
- `ＣＩＳ`

```text
顧客 -> 店舗/支店/閉店区分 -> ＣＩＳ -> 契約ID風フォルダ -> ファイル
```

#### CIS Pattern L4

```text
顧客 -> 店舗/支店/閉店区分 -> ＣＩＳ -> 作業分類 -> ファイル
```

探索順:

2. `店舗/支店/閉店` 候補を拾う
3. 派生区分候補 `現金 / リース / ＣＩＳ` を探す
4. 派生区分ごとに
   - 年フォルダがあるか
   - 契約ID風フォルダがあるか
   - 商品分類があるか
   - 作業分類があるか
   - 直接ファイル型か
   を判定する
5. 候補パスを返す

## 2. 候補パス返却の考え方

resolver は候補ごとに「絶対パスで保存可能な候補」を返す。
スコアは持たず、代わりに UI で人が判断できる説明情報を返す。

候補ごとに最低限返すもの:

- 絶対パス
- 表示用パス
- 既存フォルダかどうか
- 新規作成が必要な階層
- どのパターンに一致したか
- どの OCR 値を使ってこの候補になったか
- 注意点や警告

### 自動保存の扱い

- 自動保存は行わない
- 候補が 1 件でも必ず UI で人が確認する
- UI 上で人が保存先パスを編集できるようにする
- save API は confirm 済みの明示的な保存先だけ受け付ける

## 3. resolveSharepointDestination() の入出力仕様

### 3-1. 役割

`resolveSharepointDestination()` は OCR 結果を受けて SharePoint 上の保存先候補を解決する backend 処理。

責務:

1. OCR 抽出値の正規化
2. 顧客候補の探索
3. 業務別 resolver の実行
4. 候補パス一覧の返却

### 3-2. 入力仕様

```ts
type ResolveSharepointDestinationInput = {
  uploadId: string;
  businessType: 'COLLAB' | 'MOBILE' | 'LEASE_CASH' | 'POWER';
  siteId: string;
  driveId: string;
  baseFolderPath: string;
  ocrResult: {
    customerName?: string;
    customerNameCandidates?: string[];
    branchName?: string;
    branchNameCandidates?: string[];
    contractNumber?: string;
    contractNumberCandidates?: string[];
    documentType?: string;
    documentTypeCandidates?: string[];
    candidateOutputFileNames: string[];
  };
  namingContext?: {
    matchedRuleId?: string;
    matchedRuleName?: string;
    derivedCategory?: string;
  };
};
```

### 3-3. 出力仕様

```ts
type ResolveSharepointDestinationOutput = {
  uploadId: string;
  businessType: 'COLLAB' | 'MOBILE' | 'LEASE_CASH' | 'POWER';
  resolverName: string;
  candidates: Array<{
    candidateId: string;
    absoluteFolderPath: string;
    displayFolderPath: string;
    finalFolderPath: string;
    finalFileNames: string[];
    existingFolder: boolean;
    foldersToCreate: string[];
    matchedPattern: string;
    matchedEntities: {
      customerFolder?: string;
      branchFolder?: string;
      businessFolder?: string;
      yearFolder?: string;
      contractFolder?: string;
      workTypeFolder?: string;
    };
    derivedFrom: {
      customerName?: string;
      branchName?: string;
      contractNumber?: string;
      documentType?: string;
    };
    reasons: string[];
    warnings: string[];
  }>;
  explorationSummary: {
    scannedCustomerFolders: number;
    scannedBranchFolders: number;
    scannedBusinessFolders: number;
    fallbackSearchUsed: boolean;
  };
};
```

### 3-4. API レイヤでの返却方針

推奨 API:

```text
POST /api/uploads/:id/resolve-destination
```

返却:

- `candidates`
- `explorationSummary`

補足:

- 返却時点では保存はしない
- UI は `candidates` のいずれかを初期表示しつつ、ユーザーが絶対パスを編集できるようにする
- 候補に無いパスへ編集された場合でも、confirm 時に最終パスとして保持できるようにする

### 3-5. confirm API が受け取る確定値

```ts
type ConfirmResolvedDestinationInput = {
  candidateId?: string;
  confirmedFolderPath: string;
  confirmedAbsoluteFolderPath: string;
  confirmedFileNames: string[];
  foldersToCreate: string[];
  userConfirmed: true;
};
```

### 3-6. save API の前提

save は以下を満たさないと受け付けない。

- confirm 済み
- `confirmedFolderPath` が保存済み
- `confirmedAbsoluteFolderPath` が保存済み
- `confirmedFileNames` が保存済み
- `foldersToCreate` が保存済み

## 4. 実装順の固定案

### Step 1

共通型を作る。

- `BusinessType`
- `ResolveSharepointDestinationInput`
- `ResolveSharepointDestinationOutput`
- `ResolvedCandidate`

### Step 2

共通前処理を作る。

- 名前正規化
- かな推定
- 顧客候補探索
- 中間階層候補収集

### Step 3

4 resolver を作る。

- collabResolver
- mobileResolver
- powerResolver
- leaseCashResolver

### Step 4

resolve API を作る。

- OCR 完了後に呼ぶ
- 候補一覧を返す

### Step 5

confirm/save を分離する。

- confirm は候補確定のみ
- save は確定内容のみで保存

### Step 6

UI で絶対パスの編集を許可する。

- 候補パスの選択
- 直接編集
- 新規作成予定階層の再計算

---

## 5. 最終判断

この仕様であれば、次の理由で実装へ進める。

- resolver が見るべき階層が業務ごとに整理できた
- 探索順が固定できた
- 候補の返却形式が定義できた
- resolve API の入出力仕様が定義できた
- confirm/save 分離の前提も明確になった

したがって、次の開発は `共通型定義 -> 共通前処理 -> 4 resolver -> resolve API -> confirm/save 分離 -> UI パス編集対応` の順で着手するのが妥当
