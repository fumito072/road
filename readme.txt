あなたは、優秀なプロダクトデザイナー兼フロントエンドエンジニアです。
以下の要件に基づき、将来拡張しやすい「AIダッシュボード」のフロントエンドを実装してください。

# プロジェクト概要
このプロジェクトは第一段階として4ヶ月で進める。
最初にOCR機能を中心としたAIダッシュボードを作成する。
Vol.1のダッシュボード公開目標は4月末。
4月末にまず使い始められる状態にし、その後、実際の利用を通じて要望・フィードバックを吸い上げる。
第一段階の完了目標は6月末。
このダッシュボードはOCR専用で終わるものではなく、OCR完成後に他のAI機能や業務支援機能も順次追加していく前提なので、拡張性を最優先で設計すること。

# 会社・ブランドの前提
対象企業は、インターネット光回線や通信ネットワーク系の会社。
デザインでは以下の印象が伝わるようにすること：
- 光回線 / ネットワーク / 通信インフラ
- 安定性 / 信頼性 / 監視 / 可視化
- 先進性 / スピード / つながる感じ
- 法人向けサービスとしての安心感
- “AIっぽさ” だけでなく “ネットワーク企業らしさ” をしっかり出す

会社の世界観としては、
- 通信ネットワークをコンサルティング＆サポートする会社
- 通信に関する課題をワンストップで解決する会社
- 不便・不安・不満を、便利・安心・満足へ変える
というメッセージに合うUIにすること。

# 今回作るもの
まずは「AIダッシュボード」のトップ画面を作る。
目的は、OCR処理の状況がひと目で分かり、今後他機能を載せても破綻しないダッシュボード基盤を作ること。

# 実装の基本方針
- React + Next.js の構成を前提に実装
- TypeScript を使用
- UIは shadcn/ui もしくは再利用しやすいコンポーネント設計にする
- Tailwind CSS を使ってよい
- ダミーデータで動く状態まで作る
- 最初はフロントエンド中心でよいが、将来API接続しやすい構造にする
- コンポーネントは疎結合にし、機能追加しやすい構造にする
- デザインと実装の両方を完成度高く出すこと
- レスポンシブ対応（デスクトップ優先、タブレットでも崩れない）
- 日本語UIで作る

# デザイン要件
全体トーン：
- ベースはダークネイビー〜ディープブルー

- アクセントにシアン、ブルー、やや紫寄りの発光感を使う
- “光ファイバー”“通信回線”“ネットワーク接続”“データが流れる” イメージを背景や装飾で表現
- ただし派手すぎず、業務ダッシュボードとして見やすいこと
- SFすぎる見た目ではなく、企業向けの洗練された先進UIにする
- ガラス調、半透明、微細なグロー、細いライン、ネットワークノード風モチーフを使ってよい
- 背景にうっすらグリッドや回線の流れを感じる演出を入れてよい
- KPIカードや表は視認性最優先
- 情報の階層を明確にし、未来の機能追加にも耐えるレイアウトにする

避けたいこと：
- ただの汎用SaaSダッシュボード
- テンプレ感が強い見た目
- 未来感だけ強くて業務画面として使いにくいUI
- OCR専用に見えすぎて将来拡張できなさそうな情報設計

# 画面構成
以下のようなトップダッシュボードを作ること。

## 1. ヘッダー
- 左上にサービス名やプロダクト名が入るエリア
- サブタイトルとして「AI業務支援ダッシュボード」や「OCR / AI Workflow Dashboard」などが見える
- 右上に通知、ユーザー、設定のアイコン領域
- 将来的にワークスペース切替が置ける余白を残す

## 2. サイドナビ
サイドバーを用意し、以下のメニューを表示：
- ダッシュボード
- OCR
- 処理履歴
- フィードバック
- 分析
- 設定

さらに将来追加用として、
- AI機能
- 連携
- 管理
のプレースホルダも入れてよい

「今はOCR中心だが今後増える」ことが伝わる設計にすること。

## 3. ファーストビュー上部のサマリーカード
上部に4〜6枚程度のKPIカードを配置する。
例：
- 本日のOCR処理件数
- 処理待ち件数
- 処理成功率
- エラー件数
- 平均処理時間
- フィードバック回収数

それぞれにアイコン、数値、前日比や傾向表示を入れる。
通信・回線品質監視UIのような「健全性が見える感じ」にする。

## 4. メインエリア
以下のパネルをバランスよく配置する。

### A. OCR処理状況
- 処理中
- 完了
- エラー
- 待機中
などの状態が視覚的に分かるカードやチャート

### B. 処理推移
- 日別 / 週別のOCR処理件数推移
- 折れ線 or 棒グラフ
- 将来、OCR以外の機能も同じ枠組みで可視化できる設計にする

### C. 最新処理一覧
- ファイル名
- 受付時刻
- ステータス
- 抽出結果の要約
- 担当者
などが見えるテーブル
- 行クリックで詳細へ行けそうな設計にする

### D. エラー / 要確認パネル
- OCR失敗
- 再確認待ち
- 形式不一致
などのアラート一覧
- “運用して改善する” 前提なので、現場がすぐ気づける設計にする

### E. フィードバック収集導線
- Vol.1運用後に要望を吸い上げる前提なので、画面内に「フィードバックを送る」導線を目立ちすぎない形で置く
- サイドバーや右下固定ボタンでも可
- 「改善要望」「不具合報告」「新機能案」などが想起されるUIにする

## 5. 将来拡張のための領域
トップ画面のどこかに、
- 今後追加予定のAI機能
- 他機能モジュール
- 外部システム連携
を受けられるプレースホルダを設けること。

例：
- 「AI機能モジュール」
- 「今後追加される機能」
- 「連携サービス」
- 「分析メニュー」
などの枠

これにより、今はOCR中心でも「このダッシュボードが基盤になる」ことが見た目で伝わるようにする。

# 情報設計の考え方
最重要なのは拡張性。
今回はOCRを主役にするが、設計としては以下を守ること：
- 特定機能に依存しない共通レイアウト
- モジュール差し替えしやすいカード設計
- 分析パネルや処理パネルが機能横断で再利用できる構成
- サイドバーも機能追加に耐える
- API接続先が増えても破綻しないデータ構造
- “OCRダッシュボード”ではなく “AI業務基盤ダッシュボード” として成立させる
